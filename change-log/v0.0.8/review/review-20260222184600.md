### CRIT-01: `buildUntrackedFileDiffsWithBudget` — `remainingBudget` パラメータが Walk 内で正しく更新されない可能性
- **[code-reviewer]**: `app_devpanel_api.go` 差分行353-356
- `remainingBudget == 0` のチェックは Walk コールバックの冒頭にあるが、呼び出し元 `DevPanelWorkingDiff` で `consumedSize` を更新してから次の `buildUntrackedFileDiffsWithBudget` を呼ぶ際に、**Walk 内部では** `remainingBudget` をローカル変数(クロージャキャプチャ)として差し引いている。しかし外側の `consumedSize` は `entry.Diff` のバイト長だけで加算しており、hunk headerやヘッダー行も `Diff` フィールドに含まれるためサイズ計算自体は正しい。
- **しかし**、Walk関数内で `remainingBudget` が0になった後に `fs.SkipAll` が返されるが、`remainingBudget` が**外側のパラメータ**であるため、ディレクトリ Walk終了後に呼び出し元ループでの `budgetExceeded` フラグが正しくセットされる。この点は問題なし。
- **再検証結果**: ロジックは正しい。`remainingBudget` はクロージャで正しくキャプチャされ、`budgetExceeded` フラグで外側に伝搬される。**問題なし → Critical から降格**。

### CRIT-02: `errorLogStore.ts` — `setEntries` で `lastReadSeq` のリグレッション検出ロジック
- **[code-reviewer]**: `errorLogStore.ts` 差分行88-90
```typescript
if (maxNewSeq > 0 && maxNewSeq < lastReadSeq) {
    lastReadSeq = maxNewSeq;
}
```
- バックエンド再起動時に `seq` がリセットされるケースで `lastReadSeq` を `maxNewSeq` にリセットするのは正しい。
- **しかし**、途中でバックエンドが再起動してseqが巻き戻った場合、**既読マーカーが新しいseqの最大値にセットされるため、再起動前に未読だったエントリが暗黙的に「既読」扱いになる**。これは意図的なトレードオフだが、ユーザーが気付かない可能性がある。
- **判定**: ドキュメントにトレードオフが記載されており、CLAUDE.mdの製品仕様（開発フェーズ、後方互換不要）を考慮して **Important に降格**。

### IMP-06: `needsGitQuoting` — スペースを含むパスの Git互換クォーティング
- **[code-reviewer]**: `app_devpanel_api.go` 差分行767-774
- `strconv.Quote` を使用しているが、Git は C-style escaping（`\040` for space等）を使う。`strconv.Quote` は `\\x20` ではなくスペースをそのまま出力するため、Git標準とは微妙に異なる可能性がある。
- **テスト確認**: `TestBuildUntrackedFileDiffSingle_SpaceInPath` では `"my dir/file.txt"` の存在を確認しており、`strconv.Quote` が `"my dir/file.txt"` を出力する（スペースはそのまま）。
- **判定**: `needsGitQuoting` は `b == ' '` でtrueを返すが、`strconv.Quote("my dir/file.txt")` は `"\"my dir/file.txt\""` となりスペースはエスケープされない。**これは合成diffのヘッダーに `diff --git a/"my dir/file.txt" b/"my dir/file.txt"` が出力されることを意味し、Git自体のquoting挙動（`a/"my dir/file.txt"` → `a/my\ dir/file.txt` or `"a/my dir/file.txt"`）とは異なるが、frontendのparserで処理可能であれば問題ない。**
- **提案**: 合成diffのパスクォーティングがフロントエンドの `parseDiff` と整合するか確認推奨。テスト `TestParseWorkingDiff` の quoted header テストケースとの整合性は確認済み。

## Positive Observations (強み)

1. **セッションログの "ping + fetch" モデル移行** — イベントにペイロードを載せず、フロントエンドがfull snapshotをfetchする設計はスロットリングによるデータロスを完全に排除する。設計判断として優秀。

2. **seq ベースの識別子導入** — `ts|level|msg|source` の弱い複合キーから `uint64 seq` への移行により、フロントエンドの deduplication が確実になった。

3. **セキュリティ**: `buildUntrackedFileDiffSingleWithResolvedBase` の5段階パスバリデーション（lexical guard → Lstat → symlink check → EvalSymlinks → post-resolution guard）は堅牢。

4. **テスト品質**: フィールドカウントガードテスト、TOCTOU テスト（`fakeFileInfo`）、symlink テスト、Windows CRLF テストなど、エッジケースカバレッジが優秀。

5. **ログプレフィックスの統一** — `[DEBUG-*]` から短い `[*]` 形式への統一は可読性向上。

6. **リングバッファの堅牢化** — `cap` → `bufCap` のリネーム（組み込み関数との衝突回避）、`capacity < 1` のクランプ、`bufCap == 0` の防御は品質向上。

7. **TeeHandler のパニック回復** — `defer recover()` の追加でコールバックのパニックがslog全体を停止させない。

8. **viewerRegistry の HMR 対応** — `Symbol.for` + `subscribeRegistry` パターンで HMR 時のビュー再登録が安全に行われる。

9. **ロールバックエラーの非公開化** — `fmt.Errorf("%w (rollback failed: %v)", ...)` から `return errResp(originalErr)` に変更し、内部ロールバック詳細をクライアントに漏らさない。
